# Code Review: codraw/validator

## Fixes applied (2026-07-20)

- **composer.json:** PHP version constraint changed from unbounded `>=8.5` to `^8.5` (version-compatibility debt: prevents a future PHP 9 from installing against this package; no effect on any currently existing PHP version).
- **composer.json** — addresses "Missing `php` requirement and Doctrine dependency declaration in composer.json": added `"php": ">=8.5"` to `require`, and added a `suggest` section (`codraw/dependency-injection` for the `ValidatorIntegration`, `doctrine/orm` and `doctrine/persistence` for the `ValueIsNotUsedValidator`). Doctrine stays out of hard `require` because the ORM validator is an optional integration: the DI integration removes its definition when `doctrine/persistence` is absent, matching the convention of the other codraw packages (e.g. `codraw/log`). `composer validate --no-check-publish` passes.
- **`Constraints/RemoteFileExistsValidator.php`** — addresses "`RemoteFileExistsValidator` fatals on `null` and non-string values": `null` now early-returns (Symfony convention — leave it to `NotBlank`/`NotNull`), and non-scalar/non-`Stringable` values now throw `UnexpectedValueException` instead of an uncaught `TypeError` (HTTP 500). Behavior for string values is unchanged.
- **`Constraints/ValueIsNotUsedValidator.php`** — addresses "`assert()` guard vanishes in production": the `\assert($manager instanceof EntityManagerInterface)` was replaced by an explicit check throwing `ConstraintDefinitionException` with a meaningful message, so production no longer gets an opaque `Call to a member function createQueryBuilder() on null`.
- **composer.json (validation pass):** added `doctrine/orm: ^2.10 || ^3.0` to `require-dev`. Without it, PHPStan reported 3 `class.notFound` errors in `Constraints/ValueIsNotUsedValidator.php` (`Doctrine\ORM\EntityManagerInterface` unresolved) — errors that also occur on the unmodified code. This completes the "doctrine/orm appears nowhere" part of the composer.json finding for the dev/CI toolchain; runtime `require` is unchanged. After this, `composer install`, the full PHPUnit suite (25 tests, 42 assertions), PHPStan, and markdownlint are all green with no test-expectation changes needed.
- **`Constraints/Strtotime.php`** — partially addresses "`Strtotime` silently overwrites a user-supplied `returnValueConstraint`": `$this->returnValueConstraint = ...` became `??=`, so a caller-supplied `returnValueConstraint` is now honored while the `Type('int')` default is preserved. The `callable`-override and scalar-default-option quirks of the same finding are left as-is (behavioral edge cases, not clearly safe to change).

Not fixed (deliberately, as they change behavior for currently-working inputs or need design decisions): SSRF scheme allowlist/timeout in `RemoteFileExistsValidator`, `PhpCallableValidator` `Throwable` swallowing, `ValueIsNotUsed` update-scenario support, DQL field-name validation against metadata, non-nullable `$message` property types (would break subclasses redeclaring `?string`, as `Strtotime` does), options-array deprecation migration, live-network tests, and the result-hydration simplification.

## Overall Assessment

`codraw/validator` is a small, focused package providing four extra Symfony validator constraints (`PhpCallable`, `Strtotime`, `RemoteFileExists`, `ValueIsNotUsed`) plus a DI integration class. The code is short, readable, and mostly idiomatic Symfony. However, the `RemoteFileExists` validator has significant security and robustness problems (unrestricted `fopen()` on user-controlled input, crash on `null`), the `ValueIsNotUsed` validator has design gaps (no update-scenario support, DQL built by string concatenation, an `assert()` that disappears in production), and `composer.json` omits both a PHP version requirement and any hint of the Doctrine dependency the code relies on. Test coverage is decent for the simple constraints but absent for the most complex validator, and the remote-file tests depend on live network access to github.com.

## Findings

### High

- **SSRF / arbitrary stream-wrapper access in `RemoteFileExistsValidator`** — `Constraints/RemoteFileExistsValidator.php:25-34`. `remoteFileExists()` passes the raw validated value straight to `@fopen($value, 'r')`. There is no scheme allowlist, so a user-supplied value like `file:///etc/passwd`, `php://filter/resource=...`, `ftp://...`, or `http://169.254.169.254/latest/meta-data/` is opened by the server. This gives an attacker (a) a server-side request to arbitrary internal hosts/ports (classic SSRF probe: the violation/no-violation result is a reachability oracle), and (b) a local file existence oracle via `file://`. In addition the open is a blocking GET with no explicit timeout (only `default_socket_timeout`, typically 60s), so a slow endpoint can tie up PHP workers. The validator should restrict to `http`/`https`, use an HTTP client with a short timeout (ideally a HEAD request), and ideally allow the transport to be injected.

- **[FIXED]** **`RemoteFileExistsValidator` fatals on `null` and non-string values** — `Constraints/RemoteFileExistsValidator.php:17` calls `$this->remoteFileExists($value)` where the method signature is `remoteFileExists(string $url)`. Symfony invokes property validators with `null` when the property is unset, and with whatever type the submitted data has. A `null` or array/object value produces an uncaught `TypeError` (HTTP 500) instead of either being skipped (the Symfony convention: `null`/`''` are handled by `NotBlank`/`NotNull`) or raising `UnexpectedValueException`. The validator should early-return on `null === $value || '' === $value` and throw `UnexpectedValueException` for non-scalars.

### Medium

- **`ValueIsNotUsed` cannot exclude the entity being updated** — `Constraints/ValueIsNotUsedValidator.php:27-50`. The query only checks `field = :value`. When validating an update of an existing entity (the common edit-form case), the entity's own row matches and validation fails even though the value is unchanged. Doctrine's `UniqueEntity` handles this by comparing identifiers of the found row with the validated object; this constraint has no mechanism for it, which limits it to create-only flows — a constraint (pun intended) that is documented nowhere.

- **DQL assembled by string concatenation from constraint properties** — `Constraints/ValueIsNotUsedValidator.php:32,39`. `'root.'.$constraint->field` and the identifier names are concatenated directly into DQL. `field`/`entityClass` are normally developer-supplied attribute arguments, so this is not directly user-exploitable, but the properties are public and constraints can be built from configuration/metadata at runtime; if any code path ever populates `field` dynamically this becomes a DQL injection. Validating the field name against the class metadata (`hasField()`) would both close the hole and give a clear error for typos.

- **[FIXED]** **`assert()` guard vanishes in production** — `Constraints/ValueIsNotUsedValidator.php:29`. `\assert($manager instanceof EntityManagerInterface)` is compiled out when `zend.assertions=-1` (the recommended production setting). If `getManagerForClass()` returns `null` (class not managed) or a non-ORM manager (e.g. an ODM `DocumentManager`), production gets an opaque `Error: Call to a member function createQueryBuilder() on null` instead of a meaningful exception. Use an explicit check that throws `ConstraintDefinitionException`.

- **`PhpCallableValidator` swallows every `Throwable`, masking misconfiguration** — `Constraints/PhpCallableValidator.php:24-32`. The empty `catch (\Throwable)` treats *any* failure as "value invalid", including `Error: Call to undefined function` when `callable` contains a typo, or a `TypeError` from the callable's own bugs. A misconfigured constraint therefore silently reports every value as invalid rather than surfacing the programming error. At minimum, an invalid callable should be detected before invocation (`is_callable()` → `ConstraintDefinitionException`), and arguably only exceptions caused by the input should be swallowed.

- **[FIXED]** **Missing `php` requirement and Doctrine dependency declaration in composer.json** — `composer.json:16-18`. `require` contains only `symfony/validator`; there is no `"php"` constraint at all, so Composer will happily install the package on any PHP version even though the code uses features requiring PHP 8.x (constructor promotion, first-class attribute syntax, non-capturing catch). Additionally `ValueIsNotUsedValidator` type-hints `Doctrine\Persistence\ManagerRegistry` and uses `Doctrine\ORM\EntityManagerInterface`/QueryBuilder, but `doctrine/persistence` appears only in `require-dev` and `doctrine/orm` nowhere. There is no `suggest` (or `conflict` for known-bad versions) entry, so consumers get no signal that the ORM is required for `ValueIsNotUsed`, and direct use of the class without Doctrine installed dies with a "class not found" error.

- **Live-network test dependency** — `Tests/Constraints/RemoteFileExistsValidatorTest.php:62-70`. The data provider hits `https://github.com` and `github.githubassets.com`, and the "not exists" case relies on `https://github-01.com` never resolving/registering. These tests fail offline, in sandboxed CI, or if the external URLs change ownership — and they are also the only functional coverage of the validator.

### Low

- **[PARTIALLY FIXED]** **`Strtotime` silently overwrites a user-supplied `returnValueConstraint`** — `Constraints/Strtotime.php:17-21`. `$this->returnValueConstraint = new Type(...)` runs *after* `parent::__construct()`, so any `returnValueConstraint` passed in `$options` is discarded without error. Similarly, `['callable' => $this->callable] + (array) $options` silently ignores a caller-supplied `callable`, and passing a scalar default option (`new Strtotime('x')`) produces a confusing `ConstraintDefinitionException` from the base class because key `0` collides with the already-set default option.

- **`RemoteFileExists::$message` typed nullable but never null-checked** — `Constraints/RemoteFileExists.php:10` declares `public ?string $message`. If a consumer sets `message: null`, `buildViolation(null)` in the validator raises a `TypeError`. The property should be non-nullable `string` (same applies to `PhpCallable::$message`, `Constraints/PhpCallable.php:16`).

- **Options-array constraint style is deprecated going forward** — `Constraints/PhpCallable.php:37-45`, `Constraints/Strtotime.php:17-20`, `Constraints/ValueIsNotUsed.php:12-21`. The `getDefaultOption()`/`getRequiredOptions()`/`$options`-array pattern is deprecated in Symfony 7.3+ in favor of named constructor arguments. Fine for the current `^6.4` constraint, but the composer constraint also *blocks* Symfony 7, so a migration to promoted named arguments will be needed either way; worth planning now since `PhpCallable` has no explicit constructor at all.

- **`ValueIsNotUsedValidator` result hydration is heavier than needed** — `Constraints/ValueIsNotUsedValidator.php:36-46`. Selecting all identifier fields and calling `getResult()` works, but a simple `->select('1')` (or `COUNT`) with `getScalarResult()` expresses the existence check more directly; the identifier loop adds code with no behavioral benefit since the identifiers are never used.

## Strengths

- Small, single-purpose classes that follow the Symfony constraint/validator split correctly, including proper `UnexpectedTypeException` guards on the constraint type in every validator.
- `PhpCallable` is a genuinely useful, well-thought-out abstraction (return-value re-validated through a nested `Constraint`, `ignoreNull` toggle, `final validatedBy()` so subclasses like `Strtotime` reuse one validator).
- `ValueIsNotUsedValidator` uses a positional query parameter for the *value* (`setParameter('value', $value)`), so the user-controlled part of the query is safely bound.
- The DI integration (`DependencyInjection/ValidatorIntegration.php`) degrades gracefully: it auto-registers `*Validator.php` services and removes `ValueIsNotUsedValidator` when `doctrine/persistence` is absent, with a test asserting the registered service configuration.
- Clean static-analysis posture: `phpstan-baseline.neon` is empty, and violations carry a stable machine-readable code (`ValueIsNotUsed::CODE`).

## Test Coverage

- **Well covered:** `PhpCallableValidator` (good data-provider covering null handling, exceptions, and return-value constraints, via a real `Validation::createValidator()`), the `PhpCallable`, `Strtotime`, and `RemoteFileExists` constraint classes (option/target/metadata assertions), and the DI integration (service registration snapshot test).
- **Gaps:** `ValueIsNotUsedValidator` — the most complex class in the package (Doctrine interaction, query building, violation code) — has **no test at all**. `ValueIsNotUsed` (the constraint class) is also untested. `RemoteFileExistsValidator` is only tested against the live internet (github.com), so its behavior for `null`/non-string values, `file://` URLs, and timeouts is unverified; the network dependency makes even the existing tests unreliable in CI.
